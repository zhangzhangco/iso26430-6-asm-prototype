\SectionPage{1. 传输与安全层}{第 6 章先规定消息跑在什么样的通道里}

\begin{frame}{6.1 TLS 约束}
  \FrameAccent
  \TopTag{TLS}
  \begin{columns}[T,totalwidth=\textwidth]
    \column{0.49\textwidth}
    \begin{block}{标准要求}
      \small
      \begin{itemize}
        \item TLS 1.0
        \item 双向证书认证
        \item 唯一 cipher suite: \Code{TLS\_RSA\_WITH\_AES\_128\_CBC\_SHA}
        \item 仅支持 RSA 2048 / exponent 65537
        \item 无压缩
        \item TLS record size = 512 bytes
      \end{itemize}
    \end{block}
    \column{0.49\textwidth}
    \begin{block}{本项目对应}
      \small
      \begin{itemize}
        \item \Path{src/security/tls\_profile.cpp}
        \item \Path{src/security/certificate\_utils.cpp}
        \item \Path{tests/test\_tls.cpp}
      \end{itemize}
      \vspace{0.15cm}
      已实现：TLS 1.0、单一 cipher、禁压缩、互认证、最小 RSA 约束。\\
      \footnotesize 未实现：完整 D-Cinema 证书 profile、512-byte record、显式 ChangeCipherSpec 特殊处理。
    \end{block}
  \end{columns}
\end{frame}

\begin{frame}[fragile]{代码例子：TlsProfile 如何把条款落地}
  \FrameAccent
  \TopTag{Code}
  {\scriptsize
  \begin{block}{来自 \Path{src/security/tls\_profile.cpp}}
\begin{verbatim}
expect(SSL_CTX_set_cipher_list(ctx, "AES128-SHA"), ...);
expect(SSL_CTX_set_min_proto_version(ctx, TLS1_VERSION), ...);
expect(SSL_CTX_set_max_proto_version(ctx, TLS1_VERSION), ...);
SSL_CTX_set_options(ctx, SSL_OP_NO_COMPRESSION);
verify_mode = SSL_VERIFY_PEER | SSL_VERIFY_FAIL_IF_NO_PEER_CERT;
\end{verbatim}
  \end{block}
  }
  \vspace{0.15cm}
  \begin{block}{应该怎么理解}
    \small
    \begin{itemize}
      \item 这段代码不是“OpenSSL 调参”，而是在把标准里的通信约束硬编码成互操作基线。
      \item 这里能直接回答“项目为何能说自己部分符合 6.1”。
      \item 但它也说明：只校验到 RSA/2048/65537，不等于完整满足 D-Cinema 证书规范。
    \end{itemize}
  \end{block}
\end{frame}

\begin{frame}{6.4 RRP 约束和端口要求}
  \FrameAccent
  \TopTag{RRP}
  \begin{columns}[T,totalwidth=\textwidth]
    \column{0.48\textwidth}
    \begin{block}{标准要求}
      \small
      \begin{itemize}
        \item 使用同步 \textbf{Request-Response Pair}
        \item 同一对 SPB 之间，一个 RRP 完成后再开下一个
        \item 使用 well-known port \Code{1173}
        \item 忙时允许返回 busy
      \end{itemize}
    \end{block}
    \column{0.48\textwidth}
    \begin{block}{本项目对应}
      \small
      \begin{itemize}
        \item \Path{src/apps/asm\_responder.cpp} 默认端口 1173
        \item \Path{src/transport/tls\_server.cpp} 每次读取一条 request 再写回一条 response
        \item 当前没有真正的 busy 处理路径
      \end{itemize}
    \end{block}
  \end{columns}
  \vspace{0.25cm}
  \begin{tikzpicture}
    \node[fill=BrandInk,rounded corners=14pt,text=white,inner sep=12pt,text width=0.94\textwidth]{
      \small
      \textbf{关键理解：}ISO 26430-6 不是定义一套异步消息总线，而是定义“在 TLS socket 上做同步问答”的设备间安全控制协议。
    };
  \end{tikzpicture}
\end{frame}

\SectionPage{2. KLV 与通用元素}{6.2 与 6.3 决定每条消息长什么样}

\begin{frame}{6.2 KLV：为什么这份标准适合做字节级测试}
  \FrameAccent
  \TopTag{KLV}
  \begin{block}{标准要求}
    \small
    Request / Response 都是 KLV 包，Key 是 16-byte UL，Length 是固定 4-byte long-form BER，Value 里只放字段值。
  \end{block}
  \vspace{0.2cm}
  \begin{columns}[T,totalwidth=\textwidth]
    \column{0.54\textwidth}
    \begin{block}{本项目拆分方式}
      \small
      \begin{itemize}
        \item \Path{src/klv/klv.cpp} 负责纯 KLV 编解码
        \item \Path{src/asm/codec.cpp} 负责命令 UL 和字段拼接
        \item \Path{tests/test\_codec.cpp} 做字节级回归
      \end{itemize}
    \end{block}
    \column{0.42\textwidth}
    \begin{block}{带来的好处}
      \small
      \begin{itemize}
        \item 标准是否被破坏，一眼看十六进制就知道
        \item 互操作联调时不依赖“解释正确”
        \item 适合做黄金向量冻结
      \end{itemize}
    \end{block}
  \end{columns}
\end{frame}

\begin{frame}[fragile]{标准帧例子：GetTime Request}
  \FrameAccent
  \TopTag{Vector}
  \begin{block}{来自 \Path{docs/protocol-vectors.md}}
\begin{verbatim}
060e2b340205010102070102100000008300000400000001
\end{verbatim}
  \end{block}
  \vspace{0.15cm}
  \begin{columns}[T,totalwidth=\textwidth]
    \column{0.48\textwidth}
    \begin{block}{拆开看}
      \small
      \begin{itemize}
        \item 前 16 字节：UL，对应 GetTime Request
        \item 接着 4 字节：\Code{83 00 00 04}
        \item Value：只有 4-byte \Code{Request\_ID = 1}
      \end{itemize}
    \end{block}
    \column{0.48\textwidth}
    \begin{block}{标准意义}
      \small
      这正是 6.2 和 6.3 的结合：\\
      Key 决定命令类型，Length 决定包长，Value 放固定字段，所有多字节整数都用 big-endian。
    \end{block}
  \end{columns}
\end{frame}

\begin{frame}{6.3 通用元素：Request\_ID、Response、Batch}
  \FrameAccent
  \TopTag{Elements}
  \begin{columns}[T,totalwidth=\textwidth]
    \column{0.49\textwidth}
    \begin{block}{三件最重要的事}
      \small
      \begin{itemize}
        \item \Code{Request\_ID} 非零，Response 必须回显
        \item Response code 统一为 successful / failed / invalid / busy
        \item Batch 一律是 8-byte 头 + 固定 item 长度 + n 个 item
      \end{itemize}
    \end{block}
    \column{0.49\textwidth}
    \begin{block}{代码映射}
      \small
      \begin{itemize}
        \item \Path{include/asm/common.hpp}
        \item \Path{src/asm/codec.cpp}
        \item \Path{src/asm/event\_log.cpp}
        \item \Path{src/asm/le\_key.cpp}
      \end{itemize}
    \end{block}
  \end{columns}
  \vspace{0.2cm}
  \begin{block}{例子}
    \small
    \Code{EventIDBatch} 和 \Code{LEKeyIDBatch} 都用同一批处理思路：先写数量，再写单项长度，再写元素序列。这就是标准把“可扩展列表”做成可预测字节布局的方法。
  \end{block}
\end{frame}

\SectionPage{3. 第 7 章通用命令}{这些命令和设备类型无关，是 ASM 的基本能力}

        \item \Code{TLS profile uses development CA}

    \end{block}
  \end{columns}
  \vspace{0.2cm}
  \begin{block}{实现边界}
    \small
    \footnotesize 当前 XML 只是“对齐 Text Log Record 字段语义”的原型，不是完整 SMPTE 430-5 signed log pipeline。
  \end{block}
\end{frame}

\SectionPage{4. 第 8 章 LE 命令}{这部分把链路加密密钥管理做成一组明确 RRP}

\begin{frame}{8.1--8.5 LE 命令一览}
  \FrameAccent
  \TopTag{LEKey}
  \begin{tabular}{>{\raggedright\arraybackslash}p{0.2\textwidth} >{\raggedright\arraybackslash}p{0.25\textwidth} >{\raggedright\arraybackslash}p{0.42\textwidth}}
    \toprule
    命令 & 标准作用 & 本项目做法 \\
    \midrule
    LEKeyLoad & 下发一批 LE key & 32-byte item，支持 overflow 返回 failed \\
    LEKeyQueryID & 查某个 key 是否存在 & 返回 \Code{KeyPresent} \\
    LEKeyQueryAll & 枚举全部活动 key ID & 返回 \Code{LEKeyIDBatch} \\

    LEKeyPurgeAll & 删除全部 key & 返回最小成功响应 \\

  \end{tabular}
  \vspace{0.22cm}
  \begin{block}{关键点}
    \small
    标准并不定义真正的加密硬件实现，而是定义“管理接口的线协议”。这也是项目能用 mock key store 先把协议跑通的原因。
  \end{block}
\end{frame}

\begin{frame}{为什么 LEKeyLoad 的 batch 设计很重要}
  \FrameAccent
  \TopTag{Batch}
  \begin{columns}[T,totalwidth=\textwidth]
    \column{0.48\textwidth}
    \begin{block}{单项结构}
      \small
      \begin{itemize}
        \item LE Key ID: 4 bytes
        \item Key: 16 bytes
        \item Expire Time: 4 bytes
        \item Attribute Data: 8 bytes
      \end{itemize}
      合计正好 32 bytes。
    \end{block}
    \column{0.48\textwidth}
    \begin{block}{本项目对应}
      \small
      \begin{itemize}
        \item \Path{src/asm/le\_key.cpp} 按标准顺序拼包
        \item 支持至少 16 个 key 的容量约束
        \item 超容量时 \Code{Overflow != 0} 且 \Code{Response = failed}
      \end{itemize}
    \end{block}
  \end{columns}
  \vspace{0.2cm}
  \begin{block}{这页想说明什么}
